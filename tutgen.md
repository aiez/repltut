# tutgen.md — meta-prompt: code in, REPL tutorial out

Paste this whole file to an LLM agent, plus the inputs below.
Output: one self-contained `tut.md` — a multi-lecture course
over a codebase, narrated by numbered, machine-verified REPL
events. Scale instance of this recipe: lull's tut.md (2,700
lines, 10 lectures, 295 events, 47 glossary terms).

## 1. INPUTS (caller supplies; stop and ask if missing)

    sources:    code files to teach, in teaching order
    language:   target language + runtime + REPL command
    data:       sample datasets the examples may load; their
                env-var root; the fallback if absent
    audience:   one sentence (e.g. "grad students, basic
                programming, no Lua")
    exemplars:  kr_ch1.md = tone/pacing; fri2.md = stanza +
                numbered-REPL form; ezr.tex = coverage,
                topic order, paper dialog; Lions (cite) =
                every-line-discussed ideal; lull tut.md =
                the assembled whole. Imitate ONLY the named
                dimension of each.
    refs:       SE-principles doc, AI-principles sources,
                language quick-ref (for the appendix)
    budget:     lectures (10), blocks/lecture (4-5),
                repl-events/block (4-5) => ~200 events,
                numbered [1]> .. [N]> globally

## 2. NON-NEGOTIABLES

1. Every `[N]>` trace is REAL: generated by a replay
   harness against the actual code, pasted verbatim.
   NEVER retype or "remember" an output — every hand-copied
   trace in drafting eventually contains a fabricated cell.
   Machine-paste only; re-run and diff to verify.
2. Global event numbering, no gaps, no duplicates.
3. All code shown comes verbatim from `sources`.
4. Same seed => same trace. Reseed explicitly before any
   random block so students can reproduce mid-lecture.
5. Conceptual purity when weaving theory: attach an idea
   only where the code actually embodies it. If there is no
   code hook, leave it out and SAY you left it out.

## 3. THE HARNESS (build first, ~25 lines)

A replay script: reads a file of REPL input lines; for each,
prints `[N]> line`, evaluates it, prints real results.

    - expression lines print their values; statements print
      nothing; a trailing ";" forces statement mode (use to
      silence noisy return values, e.g. randomseed)
    - "#### title" lines pass through as block separators
    - args: input-file, start-number, preload flag (quietly
      require sources for lectures 2+)
    - print "next event: [N]" at end so the next lecture
      knows where to start

Reference implementation (Lua): `repl.lua` in this repo.
Port the idea, not the code.

Known traps (all hit in practice):
    - stderr progress output interleaves unpredictably with
      stdout: capture 2>&1 when letters are wanted in the
      trace, 2>/dev/null when not; place them after the
      triggering line when pasting
    - hash/dict iteration order may vary run-to-run
      (language hash seeds): never show unordered iteration
      of a multi-key dict in a trace; sort first
    - each lecture runs in a FRESH process: re-create needed
      state at lecture top (students must too — feature)
    - weak demos: if a trace shows "nothing happened"
      (e.g. a mean that barely moves), redesign the example
      around a comparison (treated vs random) before writing
      prose around it

## 4. WORKFLOW (two passes + checkpoints)

PASS 1 — OUTLINE. Read sources + ezr.tex-style exemplar.
Emit ONLY a one-page plan: per lecture — title, source
functions covered, blocks with event counts, acronyms
introduced. STOP for human approval. No prose before the
outline is frozen.

PASS 2 — LECTURES, one at a time, checkpoint after each:
    1. write the lecture's REPL input file (stage in a temp
       dir, e.g. /tmp/tut/ — NOT in the repo: gists are
       flat, and build intermediates are not deliverables)
    2. run the harness from the repo dir; inspect; fix weak
       or noisy traces NOW (renumbering later is expensive)
    3. write the lecture chunk around the verbatim trace

Then: glossary, language-101 appendix, assembly, audits.

## 5. LECTURE ANATOMY

    # Lecture N: Title
    intro paragraph: what, why, how it builds on N-1
    ## N.0 (optional) framing/theory section, no events
    ## N.1..N.5 lab blocks, each:
        prose stanza (claim) — kr voice: direct "you",
          admits limits, concrete before abstract
        4-space-indented source quote when the code matters
        fenced verbatim trace (4-5 events)
        at most ONE vignette (see 6)
        **Check:** one question answerable from the trace
    ## N.x woven theory sections (see 7), no events
    ## Recap — "REPL prompts covered: a-b", idea links
    **Exercises.** 3-4, reusing prompts BY NUMBER
       ("rerun [137] with k=5"); last lecture gets a capstone

## 6. ACRONYMS, VIGNETTES, GLOSSARY

Every SE/AI idea = 2-6 char acronym (WEL, ACQ, PARETO...).
    - exactly ONE vignette per acronym, at first executable
      use: blockquote, <= 8 lines, bold "ACRO — expansion."
      claim, tie to the just-run trace, primary-source ref
    - every later mention: [ACRO](#glossary) link, never a
      re-explanation
    - glossary = final table: acro | expansion | one-liner |
      first use ([L4.2](#l4)) | ref. Alphabetical.
    - links inside vignettes/glossary/appendix only; body
      prose stays link-free (kr voice)

## 7. WEAVING THE LECTURE TRACK

Lab blocks teach the code; woven sections teach the field.
Per lecture add 1-2 prose sections: motivation (lecture 1
opens with the course's falsifiable claim), theory under the
practice (CLT, NOIR, Metropolis...), interpretability +
causality (at the tree lecture), optimization history +
Pareto canon (at the population lecture), SBSE applications
+ case studies (ditto), BO/Thompson/acquisition vocabulary
(at the active-learning lecture), validation ritual (at the
classifier lecture). Each woven idea that earns an acronym
gets a vignette like any other. Apply rule 2.5 ruthlessly:
no code hook, no weave.

## 8. LANGUAGE-101 APPENDIX

A hands-on mini-tutorial of the implementation language —
NOT a reference card. Own numbered events, continuing the
global sequence; pure language, zero project/data deps, runs
in a bare REPL. Cover only constructs `sources` use; tie
each back to where the sources use it ("now re-read l.new —
that is [284]+[285] in one line"). End with a
"deliberately skipped" list. Cite the quick-ref from
`refs`.

## 9. ASSEMBLY (single flat file)

    header:  title, license, claim paragraph, setup block,
             TOC table: # | lecture | REPL range | ideas
    anchors: <a name="l1"> .. <a name="l10">, "contents",
             "glossary", "appendix" (HTML anchors — heading
             slugs are renderer-dependent; do not trust)
    back buttons: [contents](#contents) after every lecture,
             glossary, appendix; --- between sections
    order:   header, lectures 1..10, glossary, appendix

## 10. AUDITS (run all; all greppable)

    1. event numbers strictly 1..N, no gaps/dups
    2. per-lecture event count within 18-25
    3. vignette acros <-> glossary rows: exact bijection
       (extract with a digit-tolerant pattern: D2H!)
    4. every anchor referenced is defined
    5. re-run every input file through the harness; diff
       against the traces embedded in tut.md (catches
       fabrication and drift)
    6. Lions test (aspirational): every source function
       appears in some stanza; list the misses

## 11. EFFORT

Expect roughly: outline 15 min; 20-30 min per lecture
(half of it running and fixing traces); glossary + appendix
+ assembly + audits ~1 hour. Sequential total: 4-6 hours.
Checkpoint after every lecture so partial work ships.
